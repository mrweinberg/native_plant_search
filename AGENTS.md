# AGENTS.md

Notes for AI coding agents working in this repo. Optimize for the conventions already in the codebase rather than introducing new patterns.

## What this project is

A Vue 3 + Vite single-page app cataloging ~1,770 North American native plants (bedfellow.org). All data is bundled in `src/data/plants.json`; the search/list view loads a slim build-time copy (`plants-list.json`, generated by `scripts/gen-list.mjs`, gitignored) and the detail view lazy-loads the full record. No backend and no build-time *network* fetches, but the build does prerender static HTML per route and emit a sitemap (see `vite.config.js`). There is a Vitest suite (`npm test`). Deploys as static files to GitHub Pages via `.github/workflows/deploy.yml` (a `wrangler.jsonc` Cloudflare-static config also exists as an alternative).

## Conventions to preserve

### State lives in the URL, not in a store
Filter selections, search query, sort, view mode, and boolean toggles are all derived from `route.query` inside `src/composables/usePlantFilters.js`. Two declarative tables drive the filters:

- **`MULTI_FILTERS`** — array/scalar facets that produce option lists (light, soil, bloom, family, …).
- **`BOOL_FILTERS`** — boolean "only" toggles (`{ key, field, chip, icon, panelLabel, title }`); `key` doubles as the URL query param (`=1`).

The match/sort core is pure and unit-tested: `plantMatchesCriteria(plant, criteria)` and `sortPlants(list, sortBy)` (both exported). The composable just builds `criteria` from the reactive refs.

To add a **boolean** filter: add one entry to `BOOL_FILTERS`. That single entry drives the query parsing (`bools` map), the filter pass, `activeFilterCount`, the active chips in `PlantListView.vue`, and the `FilterPanel.vue` checkbox. Nothing else to wire.

To add a **multi/scalar** filter: add an entry to `MULTI_FILTERS`, add it to `FilterPanel.vue`'s `groups` array, and add a `GROUP_TITLES` entry in `PlantListView.vue` (for the chip label).

### Plants are a flat JSON array
`src/data/plants.json` is the single source of truth, committed at **1-space indent with a trailing newline**. Field schema is implicit — read existing records to learn it. Common patterns:

- Array-valued site fields (`lightRequirement`, `soilMoisture`, `soilType`, `soilPh`, `bloomMonths`, `bloomColors`, `nativeStates`, `nativeRegions`, `wildlifeValue`, `landscapeUses`) — multiple values; filters use intersection logic.
- Scalar enums (`generalAppearance`, `lifespan`, `leafArrangement`, `leafRetention`, `spreadHabit`, `family`).
- Range objects (`heightFeet`, `spreadFeet`) — `{ min, max }`.
- Booleans (`deerResistant`, `cutFlower`, `culinaryUse`, `springEphemeral`).

Categorical fields use closed vocabularies enforced by `test/data.test.js` (ENUMS / ARRAY_ENUMS) — stay inside them, and update that test when intentionally adding a value. `scripts/validate-records.mjs` gates completeness and duplicate ids.

### Adding plant records
Use the `add-plants` skill (`.claude/skills/add-plants/SKILL.md`) — it documents the full pipeline. In short: verify nativity first with `node scripts/verify-natives.mjs <names|--seed file>` (USDA L48 + GBIF, flags synonyms/dupes), then `scaffold-records.mjs` → `enrich-usda.mjs` → `enrich-county.mjs <new plants' states>` (county distribution index for the range map/county filter) → `enrich-biomes.mjs` (derives `nativeBiomes`, CEC Level I, from the county index) → `enrich-images.mjs`/`optimize-images.mjs` → editorial curation → `validate-records.mjs`. **Re-serialize `plants.json` to 1-space at the end** (`JSON.stringify(data, null, 1) + '\n'`) — the enrich scripts write 2-space, which would otherwise reformat the whole file. (`enrich-county.mjs` only writes `src/data/county-index/`, not `plants.json`.)

### Data enrichment happens in `scripts/*.mjs`
When backfilling a new field across all plants, follow the pattern in `scripts/enrich-garden-traits.mjs`: a flat object keyed by plant `id`, a writer that loads `plants.json`, merges, and writes JSON back. Print the number of records updated and any missing ids. The enrich scripts write 2-space JSON — re-normalize to 1-space before committing (see above). Don't write generated data via the build pipeline.

### Images are bundled
Plant photos live in `public/plants/<id>.jpg`, fetched and recorded by `scripts/enrich-images.mjs` (Wikimedia/iNaturalist) and downscaled by `scripts/optimize-images.mjs` (macOS `sips`). Records carry `imageFile` (+ `imageSource`/`imageCredit`, and `imageLicense`/`imageAuthor` for newer records). `usePlantImage` serves the bundled `imageFile`; the gallery (`useInatGallery`) still hits iNaturalist at runtime for the detail page's extra photos. Note: `optimize-images.mjs` recompresses *any* over-threshold file, so after running it, `git checkout -- public/plants/` to drop collateral changes and keep only your new images.

### Styling
Plain scoped CSS in each SFC, plus shared variables in `src/styles.css` (`--card`, `--border`, `--ink`, `--ink-soft`, `--accent`, `--accent-soft`, `--bg`). Use those rather than hex literals where possible. Mobile breakpoint is `max-width: 800px`.

### Routing
Web history (`createWebHistory`). Detail-page back links preserve `route.query` so the user returns to the same filtered view; the list view restores scroll position and incremental-render count on back navigation (see the `bf:listPos`/`bf:listCount` logic in `PlantListView.vue`). When adding internal navigation, pass `route.query` through.

## What NOT to do

- Don't add a state library (Pinia, Vuex). Filter state in the URL is intentional.
- Don't add a CSS framework, design tokens beyond `styles.css`, or component library.
- Don't introduce build-time *network* fetches in `vite.config.js`. Enrichment is explicit, run by a human, committed to `plants.json`. (Build-time prerendering/sitemap generation from the committed data is fine — that's what's already there.)
- Don't break the URL-as-state contract. Every filter change must be a `router.replace` of the query.
- Don't commit `plants-list.json` (gitignored; rebuilt on dev/build) or re-serialize `plants.json` to 2-space.

## Verification

```bash
npm test          # Vitest: filter logic + catalog data integrity
npm run build     # type-free build; also prerenders per-route HTML + sitemap
```

For UI changes, run `npm run dev` and verify in a browser. If you cannot run a browser, say so explicitly rather than claiming a feature works.

## Useful pointers

- `src/composables/usePlantFilters.js` — the only place filter logic lives; `plantMatchesCriteria`/`sortPlants` are the pure, tested core. Read it before touching filters.
- `src/views/PlantDetailView.vue` — companion-plants scoring lives inline at the top of `<script setup>`; weights/bonuses are documented by the code.
- `vite.config.js` — prerender (`homeContent`/`plantContent`/`aboutContent`/etc.) and sitemap generation. Keep prerendered copy in sync with the live components.
- `CATALOG_GAPS.md` and `docs/odnr-candidates.md` — ranked candidate species driving what gets added next.
- `test/data.test.js` — the catalog's enum vocabularies and integrity rules.

## When in doubt

Match the existing pattern. This codebase deliberately stays small and direct — no premature abstractions, no defensive validation of internal data beyond the committed validator/tests, no comments explaining what well-named code already says. If you're tempted to add structure "for future flexibility," don't.
